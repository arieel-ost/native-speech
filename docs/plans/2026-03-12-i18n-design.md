# Multilanguage UI (i18n) Design

**Date:** 2026-03-12
**Status:** Approved

## Goal

Add full localization to Native Speech so users who speak different native languages can use the app in their own language while learning pronunciation. Covers UI chrome, learning content, drill metadata, and AI feedback.

## Decisions

| Aspect | Decision |
|--------|----------|
| Library | `next-intl` |
| Locales | `en`, `ru`, `es`, `fr` (default: `en`) |
| URL strategy | Prefix-based (`/ru/dashboard`, `/es/practice`) |
| Detection | Auto-detect `Accept-Language` header, user can override via locale picker |
| Translation files | JSON per locale in `src/messages/`, namespaced by component |
| AI feedback | Prompt-level — instruct Gemini to respond in user's UI language |
| Mock data | Translatable strings move to JSON translation files; IDs, scores, metadata stay in TypeScript |
| Fallback | English in production, key name in development |
| Bootstrap | AI-generated initial translations for ru/es/fr, manually maintained going forward |
| Enforcement | CLAUDE.md rule (workflow) + GitHub Action (CI key-sync check) |
| Migration | Incremental: marketing → app shell → practice → auth → settings |

## Architecture

### File Structure

```
src/
├── i18n/
│   ├── routing.ts          # locale list, default locale
│   ├── navigation.ts       # locale-aware Link, redirect, useRouter
│   └── request.ts          # per-request locale config + message loading
├── proxy.ts                # Next.js 16 middleware (locale detection + redirect)
├── messages/
│   ├── en.json             # English (source of truth)
│   ├── ru.json             # Russian
│   ├── es.json             # Spanish
│   └── fr.json             # French
└── app/
    └── [locale]/
        ├── layout.tsx       # wraps with NextIntlClientProvider
        ├── (marketing)/     # public routes (home, pricing, about, auth)
        └── (app)/           # protected routes (dashboard, practice, progress, settings)
```

### Routing Configuration

`src/i18n/routing.ts`:

```typescript
import {defineRouting} from 'next-intl/routing';

export const routing = defineRouting({
  locales: ['en', 'ru', 'es', 'fr'],
  defaultLocale: 'en'
});
```

### Proxy (Middleware)

`src/proxy.ts` — Next.js 16 uses `proxy.ts` instead of `middleware.ts`:

```typescript
import createMiddleware from 'next-intl/middleware';
import {routing} from './i18n/routing';

export default createMiddleware(routing);

export const config = {
  matcher: '/((?!api|trpc|_next|_vercel|.*\\..*).*)'
};
```

- Auto-detects browser `Accept-Language` header on first visit
- Redirects to matching locale prefix
- Falls back to `en` if no match
- Excludes API routes, static assets, and Next.js internals

### Navigation

`src/i18n/navigation.ts`:

```typescript
import {createNavigation} from 'next-intl/navigation';
import {routing} from './routing';

export const {Link, redirect, usePathname, useRouter, getPathname} =
  createNavigation(routing);
```

All internal links use `Link` from `@/i18n/navigation` instead of `next/link`. This automatically prefixes the current locale.

### Request Configuration

`src/i18n/request.ts`:

```typescript
import {getRequestConfig} from 'next-intl/server';
import {hasLocale} from 'next-intl';
import {routing} from './routing';

export default getRequestConfig(async ({requestLocale}) => {
  const requested = await requestLocale;
  const locale = hasLocale(routing.locales, requested)
    ? requested
    : routing.defaultLocale;

  return {
    locale,
    messages: (await import(`../messages/${locale}.json`)).default
  };
});
```

### Layout

`src/app/[locale]/layout.tsx`:

```typescript
import {NextIntlClientProvider, hasLocale} from 'next-intl';
import {setRequestLocale} from 'next-intl/server';
import {notFound} from 'next/navigation';
import {routing} from '@/i18n/routing';

export default async function LocaleLayout({children, params}) {
  const {locale} = await params;
  if (!hasLocale(routing.locales, locale)) {
    notFound();
  }
  setRequestLocale(locale);

  return (
    <html lang={locale}>
      <body>
        <NextIntlClientProvider>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}

export function generateStaticParams() {
  return routing.locales.map((locale) => ({locale}));
}
```

## Translation File Structure

Namespaced by component, flat keys within namespaces:

```json
{
  "Metadata": {
    "title": "NativeSpeechAI — Your AI-Powered Accent Coach",
    "description": "Master any accent with AI-powered pronunciation coaching"
  },
  "Nav": {
    "features": "Features",
    "pricing": "Pricing",
    "about": "About",
    "signIn": "Sign In",
    "getStarted": "Get Started"
  },
  "Hero": {
    "headline": "Your accent has a pattern",
    "subtitle": "We decode it",
    "cta": "Start Free Assessment"
  },
  "Dashboard": {
    "greeting": "Hi, {name}",
    "score": "Score: {value}",
    "streak": "{count} day streak",
    "startPractice": "Start Practice"
  },
  "Drills": {
    "th-sounds": {
      "name": "TH Sounds",
      "description": "Master the voiceless and voiced TH"
    }
  }
}
```

**Conventions:**
- One namespace per component/page
- `{variable}` interpolation (ICU format)
- `en.json` is the source of truth — all other locales mirror its structure
- Non-translatable data (IDs, scores, dates, IPA symbols) stays in TypeScript

## Component Integration

### Using Translations

```tsx
import {useTranslations} from 'next-intl';

export function Hero() {
  const t = useTranslations('Hero');
  return (
    <>
      <h1>{t('headline')}</h1>
      <p>{t('subtitle')}</p>
      <button>{t('cta')}</button>
    </>
  );
}
```

### Locale Switcher

New component in Nav (marketing) and Header (app):
- Dropdown showing language names in their own script: English, Русский, Español, Français
- Navigates to same path with new locale prefix
- `next-intl` persists choice via `NEXT_LOCALE` cookie

### Gemini Prompt Localization

In `src/app/api/analyze/route.ts`:
- Read locale from request (header or query param)
- Append to Gemini system prompt: `"Respond entirely in {language name}"`
- No changes to audio processing — only text feedback language changes

## Migration Strategy

Incremental, one group at a time:

1. **Infrastructure** — install `next-intl`, create routing/proxy/config, move routes under `[locale]/`
2. **Marketing pages** — Hero, Nav, Footer, Features, Pricing (highest visibility)
3. **App shell** — Sidebar, Header
4. **Practice/Progress** — DrillSession, FeedbackDisplay, PhonemeTable
5. **Auth pages** — sign-in, sign-up
6. **Settings/Onboarding** — last priority
7. **Mock data migration** — move translatable drill strings to JSON files
8. **Gemini prompt localization** — add locale param to analyze API

## Enforcement

### CLAUDE.md Rule

Added to project CLAUDE.md so all future agent sessions follow:

```
## i18n — Translation Workflow

- Translations are always the LAST step of any feature implementation
- Build features with hardcoded English or translation keys first
- Once working and reviewed, extract strings to src/messages/ as the final task
- Always update ALL locale files (en, ru, es, fr) when adding keys
- Use AI-generated translations for non-English locales
- en.json is the source of truth — other locales mirror its key structure
- Use next-intl useTranslations('Namespace') hook — one namespace per component
- Use Link from @/i18n/navigation instead of next/link
- Locale switcher is in Nav (marketing) and Header (app)
```

### GitHub Action

`.github/workflows/i18n-check.yml`:
- Triggers on PR to master
- Runs `node scripts/check-i18n-keys.js`
- Recursively compares all keys in `en.json` against `ru.json`, `es.json`, `fr.json`
- **Fails** if any locale is missing keys (lists what's missing)
- **Warns** if any locale has extra keys not in `en.json`
- No external dependencies — pure Node.js script

## Fallback Behavior

- **Production:** missing translation key → show English string
- **Development:** missing translation key → show the key name (e.g., `Nav.signIn`) to make gaps obvious
- Configured via `next-intl`'s `onError` and `getMessageFallback` in request config
